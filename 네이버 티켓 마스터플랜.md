너는 내 프로젝트의 PM + 서비스 기획자 + 시니어 풀스택 아키텍트 역할을 맡는다.

나는 지금 공연 티켓/모바일티켓/좌석배정/입장관리 시스템을 만들고 있다.
이 프로젝트는 단순한 예매앱이 아니라 아래 요소가 합쳐진 서비스다.

1) 네이버 스마트스토어와 연계된 주문 기반 티켓 처리 시스템
2) 비지정석 자동 좌석배정 시스템
3) 지정석 선택 예매 시스템
4) 모바일티켓 + QR 입장 시스템
5) 현장 스캐너 및 관리자 어드민
6) 마일리지 / 친구초대 / 구독 서비스
7) 장기적으로 네이버에 판매 가능한 예매 플랫폼

내가 원하는 것은 “기획 정리”가 아니라, 실제로 개발과 운영에 바로 쓸 수 있는 수준의
구체적인 단계별 실행계획이다.

반드시 아래 조건을 지켜서 답변해라.

--------------------------------------------------
[중요한 전제]
--------------------------------------------------

- 독자 결제 시스템을 직접 만드는 것이 아니라, 당장은 네이버 스마트스토어와 연계해서 활용하는 구조가 핵심이다.
- 네이버는 접근성과 결제가 쉬우므로, 주문은 네이버를 활용하고 내 시스템은 티켓 처리, 좌석, QR, 입장, 리뷰, 구독, 운영을 담당하는 구조로 간다.
- 장기적으로는 이 시스템을 네이버 예매서비스 안에 들어갈 수 있는 수준까지 확장하는 것이 최종 목표다.
- 즉, 지금은 외부 연동형이지만 최종적으로는 판매 가능한 예매 플랫폼/SaaS 형태를 염두에 두고 설계해야 한다.

--------------------------------------------------
[현재 시스템 / 기존 자산]
--------------------------------------------------

- 현재 내 컴퓨터에는 밀론티켓 폴더와 oprncllclcl 폴더가 분리되어 있다.
- 지금 시스템은 다음과 같이 돌아간다:
  - 네이버 주문이 들어오면 텔레그램으로 알려준다.
  - 내가 승인한다.
  - 모바일 티켓 발권 시 모바일티켓을 만들어준다.
  - 놀티켓 판매 종료 이후 미판매 좌석을 올려주면 네이버 예매자들에게 좌석을 배정해준다.
  - 내가 좌석 스티커를 붙일 수 있도록 만들어주고, 봉투에 붙여서 실물티켓을 배정하는 구조다.
- 현재 파싱하고 있는 부분은 건드리면 안 된다.
- 기존 시스템을 깨지 않고 통합 운영하는 방향이 매우 중요하다.
- 즉, 신규 설계는 “기존 파싱/운영 프로세스를 최대한 보존하면서 확장”하는 방향이어야 한다.

--------------------------------------------------
[최종 목표]
--------------------------------------------------

1. 네이버 스마트스토어 연계 기반 티켓 시스템
- 독자 결제는 복잡하므로 당장은 네이버 스마트스토어를 활용한다.
- 네이버에서 주문/결제하고, 내 시스템은 티켓 발행/좌석/입장/후속 리뷰/구독을 담당한다.

2. 비지정석 좌석예매
- 놀티켓 예매가 끝난 뒤 미판매 좌석을 업데이트하면,
  예매 순서대로 좋은 좌석을 자동 배정한다.
- 배정된 좌석은 QR 코드 기반 모바일티켓에 입력된다.
- 이후 관리자가 직접 보고 수정할 수 있어야 한다.
- 공연 2시간 전에 모바일티켓을 공개한다.
- 입장할 때 스캐너로 QR을 찍고 입장 처리한다.
- 입장 스탬프가 찍힌다.
- 인터미션 때 모바일티켓으로 간단한 설문 알림을 띄운다.
- 인터미션 이후 재입장도 QR로 처리한다.
- 공연 이후에는 네이버 리뷰 이벤트 알림을 띄운다.

3. 지정석 예매
- 좌석 360도 사진 또는 좌석 시야 이미지를 업로드할 수 있어야 한다.
- 좌석을 고를 때 그 좌석에서 바라보는 무대를 확인할 수 있어야 한다.
- 네이버에서 예매 후 내 앱/웹으로 연결된다.
- 네이버 로그인 이후 해당 공연에 대해 좌석 선택 후 예매 완료가 가능해야 한다.
- 지정석은 좌석이 이미 정해져 있으므로, QR 코드만 공연 2시간 전에 공개하는 구조로 간다.
- 입장 이후 플로우는 비지정석과 동일하다.

4. 장기 목표
- 지금은 네이버와 따로 연동되지만,
  나중에는 네이버 예매서비스 안에 이 시스템을 넣을 수 있을 정도까지 발전시키는 것이 목표다.

--------------------------------------------------
[우리 서비스의 차별점]
--------------------------------------------------

- 기존 예매 → 모바일티켓 → 입장 → 리뷰까지 하나로 연결되는 구조
- PC 중심이 아니라 모바일 중심
- 공연 구독제 서비스 도입
- 좌석 클릭 시 무대 시야를 보고 예매 가능
- 사용자의 선호를 입력하면 AI가 빈 좌석 중 적절한 좌석을 추천/배정
- 실물티켓 없이 핸드폰만으로 이용 가능
- 친환경적 운영
- 공연장별 좌석맵과 시야 데이터가 축적되는 플랫폼 구조

--------------------------------------------------
[반드시 반영해야 하는 세부 요구사항]
--------------------------------------------------

1. 알림/푸시
- 예매 완료
- 공연 전 QR 공개
- 인터미션 설문
- 공연 후 리뷰 이벤트
- 취소/변경 안내
- 이 모든 알림을 보낼 수 있도록 설계해야 한다.

2. 로그인
- 현재는 URL로 모바일티켓을 받는 구조가 있다.
- 지정석 예매를 위해서는 iPhone/Android 앱 또는 모바일웹 + 로그인 구조가 필요하다.
- 네이버 스마트스토어 기반이라 네이버 로그인 중심이 적합한지 고민 중이다.
- 로그인 전략도 제안해라.
- 단, 초반에는 전환율과 운영 편의성을 고려한 현실적인 방식을 제안해라.

3. 취소처리
- 취소 종류:
  1) 취소
  2) 반품
  3) 구매확정 후 취소
- 이 상태가 감지될 때마다:
  - 원래 티켓 번호를 무효화
  - 모바일티켓 URL 삭제 또는 무효화
  - 좌석을 다시 판매 가능 상태로 전환
  - 새 예매자에게 재배정 가능하게 자동화
- 완전 자동 시스템으로 설계해야 한다.

4. 홍보 / 마일리지
- 이미 마일리지 시스템은 일부 구축되어 있다.
- 내가 준 링크로 다른 사람이 예매하면 마일리지가 적립되도록 해야 한다.
- 마일리지는 예매 시 다음 등급 업그레이드에 사용할 수 있어야 한다.
- 비행기 마일리지 같은 등급 구조를 참고하면 좋다.
- 초대한 링크로 3매 정도 구매 시 등급 업그레이드 같은 구조도 검토해라.
- 친구 초대 시 친구가 어느 좌석에 예매했는지 알 수 있도록 하는 기능도 고려해라.
- 비지정석의 경우, 친구 이름을 적어 연석 요청을 하면 따로 예매해도 같이 배정할 수 있도록 설계해라.

5. 멜팅 구독서비스
- 내 앱에 공연 등록 시 ‘구독좌석’을 별도로 떼어둘 수 있어야 한다.
- 월 구독료를 내면 공연에 대해 응모권을 주고, 추첨을 통해 좌석을 배정할 수 있어야 한다.
- 예: 월 1회는 무조건 당첨 보장 같은 정책도 검토
- 가격 정책도 현실적으로 제안해라.
- 다만 운영 손실이 나지 않도록 설계해야 한다.

6. 관리 / 안정성
- 동시에 여러 공연장에서 QR 등록을 할 수 있다.
- 한 공연장에서는 천 명 이상 입장 처리해야 하므로 오류가 치명적이다.
- 모바일티켓 오류가 나면 아무도 입장하지 못할 수 있다.
- 절대 사고가 나면 안 된다.
- 장애 대응, 백업, 오프라인/대체 처리, 긴급 수동모드까지 포함해 설계해라.
- 이 프로젝트에서 가장 중요한 가치 중 하나는 “현장 안정성”이다.

7. 좌석표 만들기
- 매우 중요한 핵심 기능이다.
- 기존에는 엑셀 좌석표를 업로드하면 외부 회사에서 좌석선택용 디자인을 만들어줬다.
- 나는 엑셀 좌석배치도를 올리면, 그것을 기반으로 좌석을 선택/구매할 수 있게 만들고 싶다.
- 지금 시스템은 인터넷 안에서 그리는 구조인데 많이 부족하다.
- PC가 아니라 모바일에서 1000석 이상 공연장을 보고도 간편하고 심플하게 좌석 선택이 가능해야 한다.
- AI 좌석 매칭 시스템을 도입해 선호 조건을 입력하면 빈 좌석 중 적절한 좌석을 추천해야 한다.
- 좌석에서 바라보는 무대 사진도 같이 보여줘야 한다.
- 모바일티켓에서 내 좌석을 클릭하면 무대배치도에서 어느 위치인지 알려주는 기능도 필요하다.

8. 공연장 등록
- 우리나라의 큰 공연장을 직접 다니며 좌석구역별 사진을 찍고 좌석배치도를 만들어 등록할 예정이다.
- 공연장별 마스터 데이터를 구축하고 싶다.
- 공연회사는 공연 등록 시 각 좌석의 좌석 등급만 설정하면 되도록 만들고 싶다.
- 등록되지 않은 공연장은 추후 직접 등록할 수밖에 없다.
- 이 과정에서 “멜팅 등록 공연장” 같은 인증/브랜딩 구조도 검토하고 싶다.

9. 어드민 사이트
- 전체 총괄은 공연장 좌석표를 제공하고,
  공연회사는 공연을 등록하고 대기 상태가 된다.
- 대표가 승인하면 그때부터 진행되도록 해야 한다.
- 어드민에서 좌석배치도를 보며,
  좌석 클릭 시 누가 예매했는지 정보가 나오고,
  다른 좌석으로 바꿀 수 있어야 한다.
- 즉, 좌석/티켓/고객/운영을 한 화면에서 통제할 수 있는 구조가 필요하다.

[답변 방식 - 매우 중요]
--------------------------------------------------

“실제로 개발을 순차적으로 진행할 수 있는 로드맵”을 만들어야 한다.

6. 단순한 아이디어 나열이 아니라,
   실제 개발자가 바로 착수할 수 있는 수준의 기획/구현 계획을 작성해라.

7. 우선순위는 반드시 다음 관점으로 판단해라.

   - 모바일티켓 안정성
   - 운영자 수동개입 가능 여부
   - 모바일 중심 사용자 경험
   - 이후 지정석/구독/마일리지/네이버 판매형 확장성

8. 너무 추상적으로 쓰지 말고,
   각 단계가 실제 기능 단위로 나뉘어야 한다.

9. 마지막에는 아래 3가지를 추가해라.
   - “가장 먼저 시작해야 할 단계 TOP 5”
   - “지금 당장 보류해도 되는 기능”

## Phase 1 — 현장 안정성 + 취소 자동화 + 운영 강화

### 1-1. 취소 자동화 파이프라인 (네이버 연동)

- **목표**: 네이버 취소/반품/구매확정후취소 → 자동으로 모바일티켓 무효화 + 좌석 해제 + 재배정 대기
- **왜 먼저**: 현재 취소가 수동이라 좌석 원장이 꼬일 위험이 가장 큼. 공연 전날 취소 폭주 시 대응 불가
- **구현 범위**:
  - Cloud Function `syncNaverCancellations`: 봇이 네이버 주문상태 변경(취소/반품/구매확정후취소)을 감지 → `cancelNaverOrderHttp` 자동 호출
  - 기존 `cancelNaverOrderInternal()`에 **취소 사유 코드** 추가 (cancel / return / post_confirm_cancel)
  - 취소 시 해당 좌석 status → `available`로 복원, mobileTicket status → `cancelled`
  - `smsTasks`에 취소 안내 SMS 자동 큐잉
  - 어드민 `NaverOrderScreen`에 취소 이력 탭 추가 (취소사유, 시각, 재배정 여부)
- **기존 시스템 연결**: `cancelNaverOrderInternal()` 이미 존재 → 사유 코드만 확장. 봇(oprncllclcl)의 polling 주기에 취소 상태 감지 로직 추가
- **주의할 점**: 이미 입장한 티켓은 취소 불가 처리 (entryCheckedInAt != null → reject). 중복 취소 방지는 기존 `allowAlreadyCancelled` 활용
- **완료 기준**: 네이버에서 취소 → 30초 내 모바일티켓 무효화 + 좌석 해제 + 취소 SMS 발송 확인

### 1-2. 취소 후 자동 재배정 시스템

- **목표**: 취소로 빈 좌석이 생기면, 대기열 순서대로 자동 재배정
- **왜 먼저**: 1-1과 세트. 취소만 되고 재배정이 안 되면 좌석이 비어서 매출 손실
- **구현 범위**:
  - Firestore trigger `onSeatStatusChange`: seat.status가 `reserved` → `available`로 변경 시 트리거
  - `waitlist` 컬렉션 신규: `{ eventId, seatGrade, naverOrderId, buyerName, buyerPhone, requestedAt, status: waiting/assigned/expired }`
  - 대기열 1순위에게 좌석 자동 배정 → mobileTicket 생성 → SMS 발송
  - 어드민에 대기열 관리 UI (순서 변경, 수동 배정, 대기열 비우기)
- **기존 시스템 연결**: `assignDeferredSeats()`의 연석 로직(`findAdjacentSeats`) 재사용
- **주의할 점**: 대기열 배정 시 race condition 방지 → Firestore transaction 필수. 대기열 만료 정책 (공연 6시간 전 자동 만료)
- **완료 기준**: 취소 발생 → 대기열 1순위 자동 배정 → SMS "좌석이 배정되었습니다" 발송

### 1-3. 현장 장애 대응 — 오프라인 모드 & 긴급 수동모드

- **목표**: Firebase/인터넷 장애 시에도 1000명 입장 처리 가능한 백업 시스템
- **왜 먼저**: 현장 사고 = 서비스 신뢰 붕괴. 한 번이라도 "입장 불가" 발생하면 치명적
- **구현 범위**:
  - **스캐너 앱 오프라인 캐시**: 공연 시작 2시간 전에 해당 이벤트의 전체 mobileTicket + ticket 목록을 로컬 SQLite/Hive에 다운로드
  - **오프라인 체크인**: 네트워크 끊김 감지 → 로컬 DB에서 QR 검증 → 네트워크 복구 시 Firestore 동기화
  - **긴급 수동모드**: 어드민에서 "긴급모드 ON" → 스캐너가 이름+전화번호 뒷자리로 수동 검색 입장 처리
  - **이중 QR 백업**: 모바일티켓에 accessToken 기반 텍스트 QR + 숫자 입장번호(entryNumber) 동시 표시 → QR 리더 고장 시 번호로 수동 입장
  - Cloud Function `downloadEventCheckSheet`: 이벤트별 전체 티켓 목록 JSON 다운로드 (스캐너 캐시용)
- **기존 시스템 연결**: 기존 `verifyAndCheckIn()`은 온라인 전용 → 오프라인은 별도 로컬 로직. 온라인 복구 시 `batchSyncCheckins` 함수로 일괄 동기화
- **주의할 점**: 오프라인 모드에서 중복 입장 방지 (로컬 DB에 체크인 플래그). 동기화 시 충돌 해결 정책 (서버 우선, but 오프라인 체크인은 유지)
- **완료 기준**: 비행기 모드에서 QR 스캔 → 입장 성공 → 네트워크 복구 후 서버에 체크인 기록 반영

### 1-4. 모바일티켓 안정성 강화

- **목표**: 모바일티켓 페이지(`/m/:accessToken`) 100% 무장애 보장
- **왜 먼저**: 모바일티켓이 안 열리면 = 입장 불가. 현재 가장 크리티컬한 단일 장애 포인트
- **구현 범위**:
  - **CDN 정적 폴백**: Vercel Edge Function에서 티켓 데이터를 렌더링한 정적 HTML을 CloudFlare CDN에 캐싱 → Firebase 장애 시에도 티켓 표시
  - **서비스 워커**: 모바일티켓 페이지에 SW 등록 → 한 번 열면 오프라인에서도 QR 표시 가능 (QR은 JWT 기반이라 로컬 생성 가능하게 개선)
  - **Health Check 알림**: Cloud Scheduler 5분 주기로 `getMobileTicketByToken` 호출 → 실패 시 텔레그램 즉시 알림
  - **에러 바운더리 강화**: 모바일티켓 화면에서 네트워크 에러 시 "마지막 로드 데이터" 캐시 표시 + 재시도 버튼
- **기존 시스템 연결**: 기존 `getMobileTicketByToken` CF + `MobileTicketPage` 위젯 확장. 기존 로직 변경 없이 캐시/폴백 레이어 추가
- **주의할 점**: QR 토큰 120초 만료 → 오프라인 시 로컬에서 재발급 가능하려면 JWT secret을 클라이언트에 내려야 하므로, 대안으로 "마지막 생성 QR 이미지 캐시" 방식 채택
- **완료 기준**: Firebase Functions 전면 장애 시뮬레이션 → 모바일티켓 여전히 QR 표시 + 스캐너 오프라인 모드에서 입장 처리 성공

### 1-5. 어드민 좌석 통합 관제 화면

- **목표**: 어드민에서 좌석배치도 위에 예매자 정보 오버레이 + 좌석 드래그 재배정
- **왜 먼저**: 현장 운영 시 "이 좌석 누구꺼?" 즉시 확인 + 좌석 변경이 핵심 운영 도구
- **구현 범위**:
  - 기존 `EventSeatManagerScreen`을 좌석맵 비주얼 모드로 확장
  - 좌석 dot 클릭 → 팝업: 예매자 이름, 전화번호 뒷4자리, 주문번호, 입장 여부, 등급
  - 드래그앤드롭 좌석 변경: 좌석 A → 좌석 B로 드래그 → `reassignTicketSeat` CF 호출
  - 필터: 등급별 / 입장완료 / 미입장 / 취소 / 비어있는 좌석
  - 실시간 Firestore 스트림으로 입장 처리 시 즉시 색상 변경 (초록=입장완료, 빨강=미입장, 회색=빈좌석)
- **기존 시스템 연결**: `seats` 컬렉션 + `mobileTickets` 컬렉션 조인. `reassignTicketSeat` CF 이미 존재
- **주의할 점**: 1000석 이상 렌더링 → Canvas 기반 (Flutter CustomPainter). 기존 `SeatLayoutEditorScreen`의 dot 좌표 재사용
- **완료 기준**: 어드민에서 좌석맵 열기 → 좌석 클릭 시 예매자 정보 즉시 표시 → 드래그로 좌석 변경 → 모바일티켓에 즉시 반영

## Phase 2 — 지정석 예매 + 좌석맵 고도화 + 로그인

### 2-1. 로그인 전략 수립 및 구현

- **목표**: 지정석 예매를 위한 최소한의 인증 체계 구축
- **왜 먼저**: 지정석 예매(2-3)의 전제조건. 로그인 없이는 좌석 선점 불가
- **구현 범위**:
  - **1차 전략: 네이버 로그인 + 전화번호 인증 (하이브리드)**
    - 네이버 스마트스토어 구매자 → 네이버 로그인으로 자연스럽게 연결
    - 기존 `signInWithNaver` CF 이미 구현됨 → 프론트 연결만 필요
    - 전화번호 인증 추가: 네이버 주문 시 입력한 전화번호로 SMS 인증 → 기존 naverOrder와 자동 매칭
  - **2차 전략: 간편 로그인 폴백**
    - 카카오 로그인 (`signInWithKakao` CF 이미 존재)
    - 구글 로그인 (Firebase Auth 기본)
    - 비회원 전화번호 인증 (지정석 예매 전용, 회원가입 없이 전화번호만으로 좌석 선택 가능)
  - **자동 주문 연동**: 로그인 시 `claimNaverOrder` CF로 기존 네이버 주문 자동 매칭 (전화번호 기준)
  - 로그인 화면 UX: "네이버로 로그인" 최상단 강조 + 카카오/구글 하단 보조
- **기존 시스템 연결**: `signInWithNaver`, `signInWithKakao` CF 이미 존재. `claimNaverOrder` CF 이미 존재. `/login` 화면 이미 존재 → 네이버 버튼 추가 + 전화번호 인증 플로우 추가
- **주의할 점**: 비지정석(네이버 직접 구매)은 로그인 불필요 유지. 지정석만 로그인 필요. 전화번호 인증은 Firebase Phone Auth 사용
- **완료 기준**: 네이버 로그인 → 기존 주문 자동 연동 → 지정석 예매 가능 상태

### 2-2. 모바일 좌석맵 UX 혁신 (1000석+ 대응)

- **목표**: 모바일에서 1000석 이상 공연장을 직관적으로 탐색 + 좌석 선택 가능한 UX
- **왜 먼저**: 지정석 예매(2-3)의 핵심 UI. 좌석을 못 고르면 지정석 의미 없음
- **구현 범위**:
  - **3단계 드릴다운 UX**:
    1. **Overview**: 공연장 전체 미니맵 (등급별 색상 구역) → 구역 탭
    2. **Zone View**: 선택 구역 확대 → 열(row) 단위 표시 → 열 탭
    3. **Seat View**: 선택 열의 좌석 가로 스크롤 → 개별 좌석 탭 → 시야 사진 팝업
  - **터치 최적화**: 좌석 터치 영역 최소 44px. 핀치 줌 + 관성 스크롤. 선택 좌석 하단 시트에 정보 표시
  - **AI 추천 모드 개선**: 기존 `SeatSelectionScreen` 3모드(AI/Zone/Quick) 유지하되, Zone 모드를 3단계 드릴다운으로 교체
  - **시야 사진 연동**: 좌석 선택 시 `venueViews` 컬렉션에서 해당 zone/floor/row의 사진 표시. 360도 뷰는 pannellum.js 웹뷰 임베드
  - **좌석 상태 실시간**: WebSocket 또는 Firestore 스트림으로 다른 사용자가 선점한 좌석 즉시 비활성화
  - **성능**: Canvas 렌더링 (CustomPainter). 뷰포트 밖 좌석은 렌더링 생략. 좌석 데이터 초기 로딩은 gzip 압축 JSON
- **기존 시스템 연결**: 기존 `SeatSelectionScreen` (5137줄) 리팩토링. `seats` 스트림 + `venueViews` 스트림 이미 존재. `VenueSeatLayout`의 free coordinates 활용
- **주의할 점**: 기존 AI/Quick 모드는 유지. Zone 모드만 교체. 좌석 선점 타임아웃 필요 (5분 홀드 → 자동 해제)
- **완료 기준**: 모바일에서 1000석 공연장 → 3초 내 로딩 → 구역 선택 → 좌석 선택 → 시야 사진 확인 → 구매 진행

### 2-3. 지정석 예매 플로우 (네이버 → 좌석 선택 → 확정)

- **목표**: 네이버에서 "지정석" 상품 구매 → 앱에서 좌석 선택 → 모바일티켓 발급
- **왜 먼저**: 비지정석만으로는 서비스 확장 한계. 지정석이 매출의 핵심
- **구현 범위**:
  - **네이버 상품 구성**: "지정석-VIP", "지정석-R", "지정석-S" 등 등급별 상품 등록 (네이버 스마트스토어)
  - **구매 후 좌석 선택 플로우**:
    1. 네이버 구매 완료 → SMS에 좌석 선택 링크 포함: `https://melonticket-web.../seats/{eventId}?naver={naverOrderId}&grade=VIP`
    2. 링크 클릭 → 네이버 로그인 or 전화번호 인증
    3. 해당 등급 좌석맵 표시 (2-2의 3단계 드릴다운)
    4. 좌석 선택 → 5분 홀드 → 확정 버튼
    5. CF `confirmDesignatedSeat`: 좌석 예약 확정 → mobileTicket 생성 (seatAssignMode: immediate)
  - **좌석 선택 기한**: 구매 후 24시간 내 좌석 미선택 시 자동 배정 (기존 `findAdjacentSeats` 로직)
  - **좌석 변경**: 확정 후 1회 변경 가능 (공연 24시간 전까지). `reassignTicketSeat` CF 확장
  - Cloud Function `confirmDesignatedSeat`: 좌석 홀드 → 확정 트랜잭션
  - Cloud Function `autoAssignExpiredDesignated`: 24시간 미선택 주문 자동 배정 (Scheduler)
- **기존 시스템 연결**: `createNaverOrder` → seatAssignMode 'designated' 추가. 기존 immediate/deferred에 이어 3번째 모드. mobileTicket 구조는 동일 (seatId 나중에 채워짐)
- **주의할 점**: 동시 선택 충돌 → Firestore transaction으로 좌석 홀드. 홀드 만료는 TTL (Firestore TTL 정책 또는 Scheduler). 네이버 취소와 좌석 선택 타이밍 충돌 주의
- **완료 기준**: 네이버 지정석 구매 → SMS 링크 → 좌석 선택 → 모바일티켓에 좌석 표시 → 공연 2시간 전 QR 공개

### 2-4. 좌석 시야 사진 + 360도 뷰 고도화

- **목표**: 좌석 선택 시 "이 좌석에서 무대가 어떻게 보이는지" 즉시 확인
- **왜 먼저**: 지정석 예매의 핵심 차별점. 시야 없이 좌석 선택은 기존 서비스와 차별 없음
- **구현 범위**:
  - 좌석 선택 시 하단 시트에 시야 사진 자동 표시 (zone/floor/row 매칭)
  - 360도 사진: pannellum.js를 Flutter WebView로 임베드 (모바일), 웹은 직접 렌더
  - 시야 사진 없는 좌석은 가장 가까운 zone의 사진으로 대체 표시 + "인근 구역 시야" 라벨
  - 모바일티켓에서 "내 좌석 위치" 탭 → 좌석맵에서 내 위치 하이라이트 + 해당 시야 사진
  - 어드민 `VenueViewUploadScreen` 개선: 드래그앤드롭 일괄 업로드 + 자동 zone/floor 태깅
- **기존 시스템 연결**: `venueViews` 컬렉션 + `VenueViewRepository` 이미 존재. `VenueSeatView` 모델에 flat/panorama180/panorama360 타입 이미 정의됨. `VenueViewUploadScreen` 이미 존재 → 확장
- **주의할 점**: 360도 사진 용량 큰 경우 → 썸네일 먼저 로드 → 탭 시 풀 로드. Firebase Storage CORS 설정 확인
- **완료 기준**: 좌석 선택 → 시야 사진 표시 (0.5초 내) + 360도 회전 가능 + 모바일티켓에서 내 좌석 위치 확인

### 2-5. 엑셀 좌석배치도 → 인터랙티브 좌석맵 자동 변환

- **목표**: 엑셀 좌석표 업로드 → 모바일 좌석 선택용 인터랙티브 맵 자동 생성
- **왜 먼저**: 현재 좌석맵은 수동 dot 에디터로 그려야 함 → 공연장마다 반복 노동. 자동화 필수
- **구현 범위**:
  - **엑셀 파싱 강화**: 기존 `ThemeColorResolver` (테마 색상 → 등급) 활용
  - **셀 좌표 → dot 좌표 자동 매핑**: 엑셀의 행/열 위치를 그대로 2D 좌표로 변환 (셀 하나 = dot 하나)
  - **자동 구역 감지**: 빈 행/열을 구역 경계로 인식 → block/zone 자동 분류
  - **스테이지 자동 감지**: 엑셀 상단의 "무대" 또는 특정 색상 영역 → stage 자동 배치
  - **미리보기 + 수정**: 자동 생성된 맵 미리보기 → 어드민에서 미세 조정 (기존 `SeatLayoutEditorScreen` 활용)
  - **저장**: `VenueSeatLayout` (v2 free coordinates)에 저장 → 좌석 선택 화면에서 바로 사용
- **기존 시스템 연결**: 기존 엑셀 파싱(ThemeColorResolver, HSL 등급분류) 100% 재사용. `SeatUploadScreen` CSV/Dotmap 탭에 "엑셀 자동변환" 탭 추가. `SeatLayoutEditorScreen` 에디터는 미세조정용으로 유지
- **주의할 점**: 엑셀 형식이 공연장마다 다를 수 있음 → 프리셋 템플릿 제공 (부산시민회관, 스카이아트홀 등). 병합 셀, 비정형 레이아웃 대응 필요
- **완료 기준**: 엑셀 업로드 → 5초 내 인터랙티브 맵 생성 → 미리보기 → 저장 → 모바일에서 좌석 선택 가능

## Phase 3 — 마일리지 / 초대 / 구독 + 알림 체계

### 3-1. 알림 파이프라인 통합 구축

- **목표**: 예매완료 → QR공개 → 인터미션 → 리뷰이벤트 → 취소안내 전체 알림 자동화
- **왜 먼저**: Phase 2의 지정석 플로우에도 알림 필요. 알림 인프라가 없으면 모든 후속 기능이 불완전
- **구현 범위**:
  - **알림 채널 3종**:
    1. SMS (기존 봇 `smsTasks` 큐 활용 — 비로그인 네이버 구매자용)
    2. FCM Push (기존 `fcmServiceProvider` 활용 — 로그인 사용자용)
    3. 인앱 알림 (새 컬렉션 `notifications` — 앱 내 알림센터)
  - **트리거 이벤트**:
    | 이벤트 | SMS | FCM | 인앱 |
    |--------|-----|-----|------|
    | 예매 완료 | ✓ | ✓ | ✓ |
    | 좌석 배정 완료 | ✓ | ✓ | ✓ |
    | QR 공개 (2시간 전) | ✓ | ✓ | ✓ |
    | 인터미션 설문 | - | ✓ | ✓ |
    | 공연 종료 리뷰 이벤트 | ✓ | ✓ | ✓ |
    | 취소/변경 안내 | ✓ | ✓ | ✓ |
    | 좌석 변경 | ✓ | ✓ | ✓ |
  - `notifications` 컬렉션: `{ userId, type, title, body, data, read, createdAt }`
  - Cloud Function `sendNotification(userId, type, data)`: 채널 자동 분기 (로그인 유저 → FCM+인앱, 비로그인 → SMS)
  - 앱에 알림센터 탭 추가 (MobileMainScreen 4번째 탭 교체 또는 벨 아이콘)
- **기존 시스템 연결**: `smsTasks` 큐 + 봇 polling 그대로 유지. `fcmServiceProvider` 이미 존재. `scheduledRevealSeats`에 QR 공개 알림 추가. `scheduledEventReminders`에 리뷰 이벤트 알림 추가
- **주의할 점**: FCM 토큰 갱신 처리. SMS 비용 관리 (건당 ~20원). 알림 빈도 제한 (동일 이벤트 1시간 내 중복 발송 금지)
- **완료 기준**: 예매 → QR공개 → 입장 → 인터미션 → 리뷰까지 전체 여정에서 알림 자동 발송 확인

### 3-2. 마일리지 시스템 고도화 + 등급제

- **목표**: 예매/리뷰/초대로 마일리지 적립 → 등급 업그레이드 → 좌석 등급 업그레이드에 사용
- **왜 먼저**: 마일리지 기본 구조(`mileageHistory`, `addMileage`, `addReviewMileage` CF)가 이미 있어서 확장 비용이 낮음
- **구현 범위**:
  - **등급 체계** (항공 마일리지 참고):
    | 등급 | 조건 | 혜택 |
    |------|------|------|
    | Bronze | 가입 즉시 | 기본 마일리지 적립 (구매액 1%) |
    | Silver | 누적 3회 예매 또는 3명 초대 | 적립 2% + 좌석 업그레이드 할인 10% |
    | Gold | 누적 10회 예매 또는 10명 초대 | 적립 3% + 업그레이드 할인 20% + 조기 좌석선택 |
    | Platinum | 누적 20회 또는 구독 6개월 | 적립 5% + 업그레이드 무료 1회/분기 + 우선 입장 |
  - **마일리지 사용처**:
    - 좌석 등급 업그레이드 (A→S: 3000P, S→R: 5000P, R→VIP: 8000P)
    - 다음 예매 할인 (1P = 1원)
  - **적립 이벤트**:
    - 예매 완료: 결제금액 × 등급비율
    - 리뷰 작성: 500P (사진포함 1000P)
    - 친구 초대 성공: 2000P
  - `AppUser` 모델에 `tier` 필드 추가 (bronze/silver/gold/platinum)
  - `users/{userId}` 에 `totalOrders`, `totalReferrals` 카운터 추가
  - Cloud Function `evaluateTierUpgrade`: 주문/초대 횟수 변경 시 등급 자동 재평가
  - 어드민 `AdminMileageScreen` 확장: 등급별 사용자 현황 + 수동 등급 변경
- **기존 시스템 연결**: `mileageHistory` 컬렉션 + `MileageRepository` 이미 존재. `addMileage`, `addReviewMileage` CF 이미 존재 → 등급비율 적용만 추가. `AppUser.referralCode` 이미 존재
- **주의할 점**: 등급 하락 정책 → 연 1회 리셋이 아닌 "유지 조건 미달 시 1등급 하락/분기" 방식. 마일리지 부정 적립 방지 (리뷰 삭제 시 마일리지 회수)
- **완료 기준**: 예매 → 자동 마일리지 적립 → 등급 변경 → 다음 예매에서 업그레이드 사용

### 3-3. 친구 초대 + 연석 배정

- **목표**: 초대 링크로 친구 예매 → 마일리지 적립 + 비지정석 연석 자동 배정
- **왜 먼저**: 마일리지(3-2)와 세트. 초대는 자연스러운 바이럴 채널
- **구현 범위**:
  - **초대 링크**: `https://melonticket.../event/{eventId}?ref={referralCode}`
  - 초대 링크로 구매 시:
    1. 초대자에게 2000P 적립
    2. 피초대자에게 1000P 웰컴 보너스
    3. 3명 초대 성공 시 등급 업그레이드 트리거
  - **연석 요청** (비지정석):
    - 예매 시 "함께 볼 친구" 입력 (이름 or 전화번호 뒷4자리)
    - `seatCompanionRequest` 컬렉션: `{ eventId, requesterOrderId, companionIdentifier, status }`
    - `assignDeferredSeats` 수정: 연석 요청이 있는 주문들을 묶어서 `findAdjacentSeats` 호출
  - **친구 좌석 확인**: 초대한 친구가 좌석 배정 후 → 초대자에게 "친구 OOO님이 S석 3열 5번에 배정되었습니다" 알림
  - `referrals` 컬렉션: `{ referrerUserId, refereeUserId, eventId, naverOrderId, status, mileageAwarded }`
- **기존 시스템 연결**: `AppUser.referralCode` + `generateUniqueReferralCode()` 이미 존재. `findAdjacentSeats()` 이미 연석 로직 보유 → 동반자 요청 반영만 추가
- **주의할 점**: 자기 초대 방지 (동일 전화번호/계정 체크). 연석은 "요청"이지 "보장"이 아님 → 불가 시 안내 메시지
- **완료 기준**: 초대 링크 공유 → 친구 구매 → 양쪽 마일리지 적립 → 비지정석 연석 배정

### 3-4. 인터미션 설문 + 공연 후 리뷰 이벤트

- **목표**: 공연 중간 설문 + 공연 후 리뷰 유도 → 네이버 리뷰로 연결
- **왜 먼저**: 현장 경험을 데이터로 전환하는 핵심 접점. 네이버 리뷰는 판매에 직접 영향
- **구현 범위**:
  - **인터미션 설문**:
    - 모바일티켓에 인터미션 체크인 후 자동 표시 (기존 intermission 스테이지 활용)
    - 간단 3문항: 만족도(별점) + 가장 좋았던 순간(객관식) + 한줄평(자유입력)
    - `intermissionSurveys` 컬렉션에 저장
    - 설문 완료 시 500P 마일리지 즉시 적립
  - **공연 후 리뷰 이벤트**:
    - 공연 종료 1시간 후 FCM/SMS 발송: "리뷰 남기고 1000P 받으세요!"
    - 앱 내 리뷰 → 기존 `Review` 모델 + `submitReview` CF 활용
    - 네이버 리뷰 유도: "네이버 리뷰도 남겨주시면 추가 500P!" + 네이버 상품 리뷰 URL 딥링크
    - 리뷰 작성 시 사진 포함하면 보너스 500P
  - Cloud Function `triggerIntermissionSurvey`: 인터미션 체크인 시 모바일티켓 화면 상태 변경
  - Cloud Function `triggerPostEventReview`: 이벤트 종료 후 스케줄러로 리뷰 알림 발송
- **기존 시스템 연결**: `intermissionCheckedInAt` 이미 존재. `Review` 모델 + `reviewRepositoryProvider` 이미 존재. `submitReview` + `addReviewMileage` CF 이미 존재
- **주의할 점**: 인터미션 설문은 30초 내 완료 가능하도록 간결하게. 네이버 리뷰 검증은 현실적으로 어려움 → 자기 신고 방식 + 스크린샷 첨부
- **완료 기준**: 인터미션 체크인 → 설문 팝업 → 완료 → 마일리지 적립. 공연 후 리뷰 알림 → 리뷰 작성 → 마일리지 적립

### 3-5. 멜팅 구독 서비스

- **목표**: 월 구독 → 공연 응모권 → 추첨 당첨 → 좌석 배정
- **왜 먼저**: Phase 3 후반. 마일리지+알림 인프라가 갖춰진 후 가능
- **구현 범위**:
  - **구독 플랜**:
    | 플랜 | 월 가격 | 응모권/월 | 당첨 보장 | 등급 |
    |------|---------|----------|----------|------|
    | Basic | 9,900원 | 2회 | 없음 | Silver 자동부여 |
    | Standard | 19,900원 | 5회 | 월 1회 보장 | Gold 자동부여 |
    | Premium | 39,900원 | 무제한 | 월 2회 보장 | Platinum 자동부여 |
  - **구독 좌석 풀**: 이벤트 생성 시 `subscriptionSeats` 필드 (등급별 N석 구독용으로 확보)
  - **응모 플로우**:
    1. 구독자가 공연에 응모 (응모권 차감)
    2. 판매 마감 3일 전 추첨 (Cloud Scheduler)
    3. 당첨자에게 구독좌석 배정 + 알림
    4. 미당첨자에게 응모권 반환 (보장형은 다음 공연 우선 배정)
  - **결제**: 네이버 스마트스토어에 "멜팅 구독 - Basic/Standard/Premium" 상품 등록 → 월 자동결제는 네이버 정기결제 API 또는 수동 월 구매 방식
  - 컬렉션: `subscriptions`, `subscriptionEntries`, `lotteryResults`
  - Cloud Function `runSubscriptionLottery`: 공연별 구독좌석 추첨
  - Cloud Function `processSubscriptionPayment`: 구독 상태 관리
- **기존 시스템 연결**: 이벤트 모델에 `subscriptionSeats` 필드 추가. 좌석 배정은 기존 `findAdjacentSeats` 재사용. 알림은 3-1 파이프라인 활용
- **주의할 점**:
  - **손실 방지 계산**: 구독좌석은 미당첨 시 일반 판매로 전환 (추첨 마감 후 잔여석 → 일반 판매). 구독 좌석 비율은 전체의 5~10%로 제한
  - Basic 9,900원으로 R석(~6만원) 당첨 가능 → 당첨 확률로 기대값 조절 (Basic: ~15% 확률)
  - 네이버 정기결제 API 미지원 시 → 매월 수동 구매 + 자동 갱신 안내 SMS
- **완료 기준**: 구독 가입 → 응모 → 추첨 → 당첨 시 좌석 배정 + 모바일티켓 발급

## Phase 4 — 공연장 마스터 데이터 + SaaS 확장

### 4-1. 공연장 마스터 데이터 구축

- **목표**: 주요 공연장의 좌석배치도 + 시야사진을 플랫폼 자산으로 구축
- **왜 먼저**: Phase 4에서 다른 공연사에 서비스를 제공하려면 공연장 데이터가 필수
- **구현 범위**:
  - **마스터 공연장 DB**: `masterVenues` 컬렉션 (플랫폼 소유, 불변)
    - 좌석배치도 (VenueSeatLayout)
    - 구역별 시야 사진 (venueViews)
    - 기본 좌석 수, 구역 구성
    - "멜팅 인증 공연장" 배지
  - **공연 등록 시**: 마스터 공연장 선택 → 등급만 설정 → 자동으로 좌석맵 + 시야 연결
  - **어드민 공연장 관리**: 마스터 공연장 등록/수정 (SuperAdmin only). 좌석배치도 에디터 + 시야 사진 업로드 통합 화면
  - **인증 공연장 뱃지**: 앱에서 "멜팅 인증 공연장" 라벨 → 시야 사진 보장 표시
  - 초기 타겟: 부산시민회관, 스카이아트홀, 세종문화회관, 예술의전당, 블루스퀘어 (5개소)
- **기존 시스템 연결**: `venues` 컬렉션 + `VenueRepository` 이미 존재. `VenueSeatLayout` v2 이미 존재. `VenueViewUploadScreen` 이미 존재. 부산시민회관/스카이아트홀 프리셋 이미 존재
- **주의할 점**: 마스터 데이터는 이벤트별로 복사하지 않고 참조 방식 (venueId로 연결). 공연사가 등급을 변경해도 마스터 좌석배치는 불변
- **완료 기준**: 5개 공연장 마스터 등록 → 공연 등록 시 공연장 선택만으로 좌석맵 자동 연결

### 4-2. 공연사(Seller) 셀프서비스 포털

- **목표**: 공연사가 직접 공연 등록 → 승인 → 판매 → 정산까지 셀프서비스
- **왜 먼저**: SaaS 확장의 핵심. 대표가 모든 공연을 직접 등록하는 건 확장 불가
- **구현 범위**:
  - **공연사 가입**: 사업자등록번호 + 대표자 정보 → 승인 요청 (기존 `adminApprovalRequests`)
  - **공연 등록 워크플로**:
    1. 공연사가 공연 등록 (draft)
    2. 마스터 공연장 선택 → 등급 설정 → 가격 설정
    3. 네이버 스마트스토어 상품 URL 연결 (또는 직접 등록 가이드)
    4. 대표(SuperAdmin) 승인 → active
    5. 판매 시작
  - **정산 자동화**: `settlements` + `escrowAccounts` 이미 존재 → 공연 종료 후 자동 정산 계산
  - **공연사 대시보드**: 내 공연 목록 + 판매 현황 + 정산 내역 + 체크인 현황
  - **수수료 정책**: 티켓 가격의 5~8% (등급별 차등, 구독서비스 공연은 3%)
- **기존 시스템 연결**: `isSeller` 역할 + `SellerProfile` 이미 존재. `SettlementRepository` + `SettlementScreen` 이미 존재. `eventCreateScreen` 이미 존재 → seller 접근 시 제한된 버전 표시
- **주의할 점**: 공연사는 다른 공연사의 데이터 접근 불가 (eventId 기반 필터 + Firestore 보안 규칙). 정산은 공연 종료 + 환불 기간(7일) 후 확정
- **완료 기준**: 공연사 가입 → 공연 등록 → 대표 승인 → 판매 → 정산 확인까지 전체 플로우

### 4-3. 네이버 예매 서비스 연동 준비

- **목표**: 네이버 예매 서비스 입점을 위한 API 표준화 + 데이터 정합성 강화
- **왜 먼저**: 최종 목표. Phase 1~4-2가 완료된 후 착수
- **구현 범위**:
  - **API 표준화**: REST API 문서 작성 (OpenAPI 3.0)
    - 공연 조회/등록/수정
    - 좌석 조회/예약/확정/취소
    - 티켓 발급/검증/입장
    - 정산 조회
  - **데이터 정합성**: 좌석 원장 감사 로그 (`seatAuditLog` 컬렉션) — 모든 좌석 상태 변경 추적
  - **SLA 목표 설정**:
    - API 응답시간 P95 < 500ms
    - 가동률 99.9%
    - 동시 예매 처리 1000 TPS
  - **보안 강화**: API 키 관리, Rate limiting, IP 화이트리스트
  - **테스트 환경**: 스테이징 Firebase 프로젝트 별도 구축
- **기존 시스템 연결**: 기존 HTTP 함수들(`createNaverOrderHttp`, `listEventsHttp` 등)을 표준 REST API로 리팩토링
- **주의할 점**: 네이버 예매 서비스 실제 입점 요건은 별도 확인 필요. API 표준화는 다른 플랫폼(카카오, 인터파크 등) 연동에도 활용 가능하도록 범용 설계
- **완료 기준**: OpenAPI 문서 완성 + 스테이징 환경 구축 + 부하 테스트 1000 TPS 통과

## 기존 시스템을 유지하면서 통합하기 위한 핵심 원칙

### 원칙 1: "기존 파이프라인은 건드리지 않고, 옆에 붙인다"
- 현재 동작하는 봇(oprncllclcl) → 텔레그램 알림 → 승인 → 발권 흐름은 그대로 유지
- 새 기능은 기존 컬렉션에 필드 추가 or 새 컬렉션으로 확장
- 절대로 기존 Cloud Function의 시그니처를 변경하지 않음 (하위 호환)

### 원칙 2: "컬렉션 스키마는 추가만, 삭제/변경은 금지"
- `mobileTickets`, `naverOrders`, `seats` 등 기존 필드는 건드리지 않음
- 새 필드는 optional로 추가 (기존 문서에 없어도 에러 안 나게)
- enum 값은 추가만 (예: seatAssignMode에 'designated' 추가)

### 원칙 3: "봇 API는 그대로, 새 엔드포인트는 별도"
- `createNaverOrderHttp`, `listEventsHttp` 등 봇이 사용하는 API는 시그니처 유지
- 새 기능은 별도 엔드포인트 (예: `confirmDesignatedSeatHttp`)
- BOT_API_KEY 인증 체계 그대로 유지

### 원칙 4: "좌석 원장(seats 컬렉션)은 single source of truth"
- 좌석 상태는 반드시 `seats` 컬렉션만이 권위 있는 원본
- mobileTicket.seatId, ticket.seatId는 seats 문서 참조
- 좌석 상태 변경은 반드시 Firestore transaction 내에서만

### 원칙 5: "장애 시 수동 개입 가능하도록 설계"
- 모든 자동화에는 어드민 수동 오버라이드 버튼 동반
- 자동 배정 실패 → 수동 배정 UI로 폴백
- 자동 취소 실패 → 어드민 알림 + 수동 취소 버튼
- 봇 장애 → 어드민에서 직접 주문 생성/취소 가능 (이미 가능)

### Claude — 기존 시스템 안전 운영 원칙 (2026.03.17)

> 🔴 **핵심 약속: 기존 시스템은 절대 깨뜨리지 않는다**

현재 운영 중인 파이프라인(네이버 주문 → 텔레그램 → 승인 → 발권 → 좌석배정)이 가장 중요합니다.
새 기능 개발 시 아래 원칙을 반드시 지킵니다:

**1. 격리 개발 (Isolation)**
- 새 기능은 기존 코드 수정 없이 **별도 파일/함수로 추가**
- 기존 Cloud Function 시그니처 변경 금지
- 기존 Firestore 컬렉션 필드 삭제/이름변경 금지 (추가만)

**2. 내부 테스트 전용 운영**
- 새 기능은 비공개 상태로 개발 + 내부 테스트

- Feature Flag 방식: 이벤트별 `enableNewFeatures: true` 플래그로 새 기능 ON/OFF

**3. 롤백 보장**
- 모든 새 기능은 Feature Flag로 즉시 OFF 가능
- Cloud Functions 배포 시 기존 함수는 건드리지 않고 새 함수만 추가
- 문제 발생 시 새 함수만 삭제하면 원상복구

**4. 배포 안전망**
- Cloud Functions 배포: `--only functions:새함수이름` (전체 배포 금지)
- Vercel 배포: git push 기반이므로 revert commit으로 즉시 롤백
- Firebase Hosting(admin): 동일하게 이전 버전으로 롤백 가능

**5. 테스트 플로우**
- 개발 완료 → 대표가 직접 테스트 → 피드백 → 수정 → 재테스트 → 승인 → 공개

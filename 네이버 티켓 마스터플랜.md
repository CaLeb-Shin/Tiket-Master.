# 네이버 티켓 마스터플랜

> 기준일: 2026-03-07

## 문서 목적

이 문서는 대화 로그가 아니라, 앞으로 네이버 티켓 관련 의사결정과 개발
우선순위를 정하는 **기준 문서**다.

핵심은 2가지다.

1. 지금 당장 운영 가능한 **모바일 티켓 발권 시스템**을 완성한다.
2. 나중에는 **네이버 로그인 + 좌석 선택형 예매 플랫폼**으로 확장한다.

---

## 최종 비전

네이버 스마트스토어에서 공연을 판매하되, 실제 발권 경험과 현장 입장 경험은
우리 시스템이 책임진다.

사용자는 구매 후 예쁜 모바일 티켓을 받아 카카오톡으로 쉽게 전달하고,
공연장에서는 승인된 기기로 QR을 스캔해 입장한다.

장기적으로는 네이버 구매 이력과 우리 앱을 연결해, 사용자가 좌석까지 직접
고르고 예매할 수 있는 구조로 확장한다.

---

## 제품 트랙

### 트랙 A. 놀티켓 연계형 모바일 티켓 발권

**정의**
- 현재 운영을 가장 빨리 바꿀 수 있는 실전형 시스템
- 네이버 스마트스토어 판매는 유지
- 놀티켓과의 연결은 유지하되, 종이 봉투 작업과 높은 수수료 의존도를 줄임
- 구매자에게는 모바일 티켓 URL을 발급

**현재 이미 있는 기반**
- 네이버 스마트스토어 구매 감지
- 텔레그램 알림
- 뿌리오 문자 발송
- 미판매 좌석 엑셀 기반 우선순위 좌석 배치
- 좌석 배치 결과를 운영자가 확인할 수 있는 흐름

**지금 없애고 싶은 운영 방식**
- 놀티켓 티켓 도착 후 이름 적고 봉투에 넣는 수작업
- 현장 종이 티켓 확인
- 구매자 전달 경험이 예쁘지 않고 공유성이 떨어지는 구조

**트랙 A의 최종 상태**
- 구매 직후 또는 운영 승인 후 모바일 티켓 URL 발급
- 공연 시작 2시간 전에 좌석과 QR 동시 공개
- 입장 때만 QR 스캔
- 인터미션은 QR 재검표보다 화면 제시 확인 중심
- 승인된 현장 기기만 체크인 가능
- 카카오톡 공유와 인스타 업로드에 적합한 티켓 디자인

**트랙 A의 운영 타임라인**
1. 네이버 스마트스토어에서 구매 발생
2. 텔레그램 알림과 문자 발송
3. 놀티켓 마감 시점에 미판매 좌석 엑셀 수신
4. 운영자가 좌석 데이터 확인 또는 업로드
5. 시스템이 우선순위 좌석으로 자동 배정
6. 모바일 티켓 URL 발급 및 전달
7. 공연 시작 2시간 전에 좌석과 QR 공개
8. 현장 입장 시 승인된 기기로 QR 체크인

**트랙 A의 비목표 (현 단계)**
- 사용자가 직접 좌석을 고르는 기능
- 놀티켓을 즉시 완전히 제거하는 것
- 환불/변경까지 전부 우리 시스템으로 먼저 옮기는 것
- 현장 운영 전부를 모바일 앱 하나로 단번에 통합하는 것

### 트랙 B. 네이버 예매 + 좌석 선택형 자체 플랫폼

**정의**
- 장기 확장형 시스템
- 네이버 로그인 또는 네이버 계정 연동 기반
- 사용자가 우리 앱/웹에서 공연과 좌석을 직접 고르고 예매

**트랙 B의 최종 상태**
- 네이버 계정으로 간단히 진입
- 구매 이력과 공연 정보 자동 연동
- 좌석 선택 가능
- 좌석별 실제 시야 사진 제공
- 어르신도 쉽게 이해하는 단순한 UX
- 네이버의 환불/구매 장점은 유지

## 지금부터의 원칙

- 트랙 A를 먼저 끝낸다.
- 트랙 A에서 실제 공연 운영이 가능해질 때까지 트랙 B를 크게 벌리지 않는다.
- 모든 정책은 문서와 코드에서 같은 말만 하게 맞춘다.
- 사용자 경험은 항상 `문자 수신자`, `카카오톡 전달받는 사람`, `현장 입장객`
  기준으로 본다.
- 고급스럽게 보이되, 이해하기 어렵게 만들지 않는다.
- 나중에 브랜드명을 바꿀 가능성을 열어두고 구조를 만든다.

## 트랙 A 상세 목표

### A-1. 발권 경험 완성

- 네이버 주문이 들어오면 MobileTicket이 생성된다.
- 구매자에게 모바일 티켓 URL이 전달된다.
- 그룹 구매자는 메인 화면에서 전체 티켓을 보고 개별 전달할 수 있다.
- 받는 사람 이름을 지정해서 전달할 수 있다.
- 카카오톡 미리보기와 공유 문구가 보기 좋게 노출된다.
- 이미지 저장 기능으로 인스타 업로드가 쉬워진다.

### A-2. 좌석 공개 정책 완성

- 좌석 공개 시점은 **공연 시작 2시간 전**으로 통일한다.
- QR 공개 시점도 **공연 시작 2시간 전**으로 통일한다.
- 공개 전에는 등급, 예매자 정보, 공연 정보만 보여준다.
- 공개 후에는 좌석, QR, 상태 정보가 명확하게 보인다.

### A-3. 현장 체크인 운영 완성

- 승인된 스캐너 기기만 입장 처리가 가능해야 한다.
- 체크인은 입장 시점에만 QR 스캔으로 처리한다.
- 인터미션은 재스캔보다 티켓 화면 확인을 기본 운영 정책으로 둔다.
- 취소됨, 이미 사용됨, 유효함 상태가 현장에서 한눈에 구분돼야 한다.

### A-4. 운영 자동화 완성

- 텔레그램 알림에서 끝나지 않고, 발권과 전달까지 이어진다.
- 공연 마감 시점, 좌석 배정 시점, 좌석 공개 시점을 운영자가 제어할 수 있다.
- 취소, 예외 좌석 이동, 재배정, 수동 보정 흐름이 운영 도구에 있어야 한다.
- 정산으로 이어질 수 있도록 주문/발권/입장 데이터가 정리돼야 한다.

### A-트랙 완료 기준

- 종이 봉투 작업 없이 모바일 티켓만으로 운영 가능
- 공연 시작 2시간 전에 좌석/QR 공개가 정확히 동작
- 현장에서 승인된 기기로만 체크인 가능
- 그룹 티켓 전달 흐름이 자연스럽고 헷갈리지 않음
- 카카오톡과 인스타 기준으로 티켓 공유 품질이 만족스러움

## 트랙 B 상세 목표

### B-1. 계정 연동 기반

- 네이버 로그인 도입 또는 현재 로그인 체계와 네이버 계정 연동
- 네이버 구매 이력과 사용자 계정 매핑
- 공연, 회차, 주문, 좌석 선택 데이터 모델 정리

### B-2. 좌석 선택형 예매 UX

- 공연 선택
- 회차 선택
- 좌석 선택
- 예매 완료

이 4단계가 최대한 단순하게 이어져야 한다.

### B-3. 좌석 시야 경험

- 좌석별 또는 구역별 실제 사진 제공
- 사용자가 "이 자리에서 어떻게 보이는지"를 직관적으로 이해
- 시야 정보는 미적 요소가 아니라 예매 결정을 돕는 기능으로 설계

### B-4. 플랫폼 확장

- 네이버 판매와 자체 예매를 병행 가능한 구조 설계
- 환불, 변경, 정산, 프로모션까지 운영 범위 확장
- 멜론티켓 명칭 유지 여부 검토

### B-트랙 완료 기준

- 사용자가 네이버 계정으로 진입해 공연을 찾고 좌석을 고를 수 있음
- 좌석 시야 정보가 실제 선택에 도움이 됨
- 예매와 환불 흐름이 쉬움
- 운영자가 관리 가능한 수준으로 시스템이 정리됨

## 디자인 목표

### 티켓 디자인

- 티켓은 예뻐야 한다.
- 단순히 예쁜 수준이 아니라, 인스타에 올려도 홍보물처럼 보여야 한다.
- 예매자 이름, 받는 사람 이름, 공연명, 공연 포스터가 핵심 시각 요소다.
- 지나치게 복잡한 효과보다 품격 있고 안정적인 레이아웃이 중요하다.

### 공유 경험

- 카카오톡에 붙여넣으면 공연 포스터, 공연명, 장소, 일시가 잘 보여야 한다.
- 전달받은 사람이 링크만 눌러도 이해할 수 있어야 한다.
- 이미지 저장과 공유가 쉬워야 한다.

### 고령층 친화성

- 버튼은 크고 명확해야 한다.
- 용어는 어렵지 않아야 한다.
- 한 화면에서 해야 할 행동이 하나씩 보이게 설계한다.
- 좌석 선택형 시스템에서도 복잡한 조작을 최소화한다.

### 브랜드 방향

- 현재는 멜론티켓 기반으로 간다.
- 다만 장기적으로 브랜드명을 바꿔도 구조가 흔들리지 않게 설계한다.
- 시각 톤은 고급스럽고 차분해야 하며, 공연/티켓 브랜드답게 보여야 한다.

## 기술 방향

### 현재 기준 권장 구조

- 판매/주문 감지: 네이버 스마트스토어 + 기존 봇
- 운영 트리거: 텔레그램 + 어드민
- 발권/정책/상태: Firebase Cloud Functions + Firestore
- 공개 티켓: Flutter Web 공개 링크
- 현장 체크인: 승인된 스캐너 기기 + QR 검증

### 시스템 책임 분리

- 네이버 스마트스토어:
  결제, 구매 채널, 네이버 기반 주문 경험의 출발점
- 놀티켓:
  트랙 A 단계에서는 미판매 좌석 정보와 운영 연계 파트의 일부를 담당
- 멜론티켓:
  모바일 티켓 발권, 티켓 전달, 좌석/QR 공개, 현장 체크인, 티켓 디자인 경험 담당
- 어드민:
  공연별 운영 제어, 예외 처리, 좌석 업로드, 취소/상태 관리 담당

### 반드시 맞춰야 할 정책

- 공개 시점 문서/코드 통일
- 주문 취소 정책 문서/코드 통일
- 그룹 티켓 전달 정책 문서/코드 통일
- 운영 승인 가능한 기기 정책 문서/코드 통일

### 지금 꼭 먼저 확정해야 할 운영 정책

- 모바일 티켓 URL 발급 시점:
  구매 직후인지, 놀티켓 마감 후 좌석 배정 완료 시점인지
- 좌석 배정 트리거:
  자동 실행인지, 운영자 확인 후 실행인지
- 인터미션 재입장 정책:
  무조건 화면 확인인지, 특정 공연은 재스캔인지
- 취소 연동 기준:
  어느 시스템의 상태를 최종 truth로 볼지
- 예외 좌석 처리:
  수동 이동, 삭제, 좌석 비공개 상황을 누가 어떻게 처리할지

### 트랙 B를 위해 미리 열어둘 것

- 네이버 계정 연동 확장성
- 좌석 선택 데이터 구조
- 시야 사진 연결 구조
- 브랜드명 변경 가능성

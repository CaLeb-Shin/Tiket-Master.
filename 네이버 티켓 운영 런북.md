# 네이버 티켓 운영 런북

작성일: 2026-03-08
범위: Ralph `MT-066` ~ `MT-075`

## 1. 트랙 A 실운영 체크리스트

### 공연 전날
- 네이버 주문 수집 완료 여부 확인
- 놀티켓 마감 시점과 좌석 배정 시작 시점 확인
- 자동 좌석 배정 trigger 기준:
  - 놀티켓 마감 완료
  - 미판매 좌석 엑셀 업로드 완료
  - 해당 시점을 `좌석 배정 시작`으로 기록
- `Naver Orders` 화면에서 `공개 상태`, `전달 현황`, `문자 현황` 최종 점검
- `앱 미연결` 주문이 많은 경우, 카카오톡/문자 안내 문구에 모바일 티켓 링크와 주문 연결 가이드를 같이 발송
- 스캐너 기기 승인 상태와 배터리, 네트워크 상태 확인

### 공연 당일 오픈 전
- 좌석 배정 완료 여부 확인
- SMS 실패 주문 재확인
- 그룹 티켓, 전달 완료 티켓, 미전달 티켓 비율 확인
- `revealAt`이 공연 시작 2시간 전으로 설정되었는지 확인
- 현장 스태프에게 `입장 시 QR`, `인터미션은 화면 확인 중심` 정책 재전달

### 공연 시작 2시간 전
- 좌석/QR 자동 공개 여부 확인
- 공개 전 상태가 남아 있으면 `좌석 공개` 버튼으로 수동 공개
- 대표 테스트 링크 1개를 열어 좌석, 이름, QR 표시 상태 확인
- 카카오톡 공유 미리보기와 인스타 저장 이미지 확인

### 입장 시작 직전
- 스캐너를 `입장 확인` 모드로 고정
- 취소 주문, 재배정 주문, VIP 현장 응대 목록 최종 확인
- 모바일 티켓 상태가 `사용 가능`으로 보이는지 확인

### 인터미션
- 기본 원칙은 화면 확인
- 재입장 기록이 필요한 경우에만 QR 재스캔
- 스캐너는 `인터미션 확인` 모드로 전환

### 공연 종료 후
- 사용 완료/입장 완료/취소됨 수량 확인
- 미사용 티켓, 전달 실패 티켓, 현장 수동 처리 건 기록
- 정산용 주문 수, 발권 수, 입장 수 스냅샷 저장

## 2. 트랙 A 예외 대응표

| 상황 | 1차 대응 | 2차 대응 | 최종 기록 |
| --- | --- | --- | --- |
| SMS 발송 실패 | 어드민 문자 현황 확인 | 티켓 링크 수동 전달 | progress 기록 |
| 좌석 공개 누락 | `좌석 공개` 버튼 즉시 실행 | 대표 링크로 검증 | 운영 로그 |
| 전달 이름 오입력 | 수신자 이름 재입력 | 새 링크 공유 | 전달 이력 메모 |
| 좌석 재배정 필요 | 주문 확장에서 재배정 | 현장 안내 재공지 | `seatReassignHistory` 확인 |
| 스캐너 미승인 | 승인된 기기로 교체 | 관리자 승인 | 기기 목록 점검 |
| 앱 미연결 주문 | `네이버 주문 연결` 안내 | 주문번호+연락처로 사용자 직접 연결 | 주문 문서 `userId` 확인 |

## 3. 스태프용 한 장 가이드

- 입장 시작 전: 스캐너 승인 상태 확인, `입장 확인` 모드 유지
- QR이 안 열리면: 공개 전 여부, 취소 여부, 사용 완료 여부를 먼저 확인
- 인터미션: 기본은 화면 확인, 꼭 필요한 경우만 QR 재스캔
- 이상 케이스: 재배정, 미전달, 문자 미수신은 운영자에게 즉시 전달
- 현장 멘트 기준:
  - QR이 안 보이면: `공연 시작 2시간 전부터 QR이 열립니다.`
  - 인터미션이면: `지금은 화면만 보여주셔도 됩니다.`
  - 주문 연결 문의면: `앱에서 주문번호와 연락처로 네이버 주문 연결이 가능합니다.`

## 4. 트랙 A Go / No-Go 기준

### Go
- 좌석 배정률 100%
- 공개 시점 이후 대표 링크 3건 이상 정상 표시
- 스캐너 승인 기기 2대 이상 준비
- 문자 실패 주문이 전체의 5% 이하
- 운영자가 `OPERATIONS` 화면만으로 상태 판단 가능

### No-Go
- 공개 후에도 `공개 전` 티켓 다수 존재
- 스캐너 승인 기기 부재
- 주문/티켓/좌석 상태가 서로 다른 기준으로 보임
- 공연 당일 재배정 정책이 정리되지 않음

## 5. 트랙 B 계정 연동 기준

### 사용자 식별 기준
- 네이버 직접 로그인: Firebase UID `naver:{naverId}`
- 기존 앱 사용자: 현재 계정 유지
- 네이버 주문 연결: `naverOrders.userId`에 현재 사용자 UID 기록

### 연결 정책
- 주문 연결은 `주문번호 + 연락처 확인`으로 수행
- 이미 다른 계정에 연결된 주문은 재연결 불가
- 같은 계정이 다시 연결하면 idempotent 성공 처리
- 연결 해제는 운영자 승인 후 처리하는 정책으로 시작

### 중복 방지 기준
- 네이버 로그인은 `naverId` 기반 고유 UID로 충돌 방지
- 주문 연결은 `naverOrders.userId` 단일 필드로 소유 계정 고정

## 6. 트랙 B 주문 매핑 기준

- 사용자 앱: `네이버 예매 내역` 화면에서 연결된 주문 조회
- 예외 화면: 연결된 주문이 없으면 빈 상태 + `주문 연결` 버튼
- 어드민: 주문 목록에서 `앱 연결됨 / 앱 미연결` 배지 확인
- 연결 소스 기록: `linkSource = selfClaim`

## 7. 좌석 선택형 예매 데이터 모델 기준

### 핵심 상태 용어
- 주문: `pending`, `paid`, `failed`, `refunded`, `canceled`
- 좌석: `available`, `reserved`
- 좌석 유형: `normal`, `wheelchair`, `reserved_hold`
- 모바일 티켓: `active`, `cancelled`, `used`

### 이행 전략
- 트랙 A: `naverOrders`, `mobileTickets` 중심
- 트랙 B: `orders`, `tickets`, `seatBlocks` 중심
- 공통 truth:
  - 좌석 재고는 `events.availableSeats`
  - 입장 상태는 `tickets/mobileTickets` 체크인 기록
  - 좌석 재배정 이력은 티켓 문서에 누적

## 8. 좌석 선택형 UX 기준

### 현재 4단계
1. 공연 선택
2. 회차/좌석 선택
3. 결제 확인
4. 예매 완료

### UX 원칙
- 모바일 기준 큰 버튼
- 한 화면에 한 행동
- 좌석, 금액, 회차 정보를 마지막 확인까지 유지
- 용어는 `좌석 선택`, `결제 확인`, `예매 완료`처럼 단순하게 유지

## 9. 결제/좌석 확정 기준

- `createOrder`는 주문을 `pending`으로 생성
- `confirmPaymentAndAssignSeats` 트랜잭션에서 좌석 확정과 주문 상태 변경을 함께 처리
- 결제 실패 시 주문은 `failed`
- 중복 좌석 점유는 트랜잭션 기준으로 차단
- 결제 성공 후 주문/티켓/좌석 수량은 같은 시점에 기록

## 10. 시야 사진/파노라마 운영 기준

- 좌석 또는 구역 대표 이미지가 있으면 예매 화면과 티켓 상세에 노출
- 이미지가 없으면 플레이스홀더와 안내 문구 노출
- `venue_view_upload_screen.dart`를 관리 기준 화면으로 사용
- 모바일 성능 기준:
  - 기본은 대표 이미지 우선
  - 180°/360°는 사용자 액션 이후 열기

## 11. 환불/변경/정산 Truth Table

| 항목 | 최종 truth |
| --- | --- |
| 네이버 주문 상태 | `naverOrders.status` |
| 일반 예매 주문 상태 | `orders.status` |
| 모바일 티켓 상태 | `mobileTickets.status`, `entryCheckedInAt`, `usedAt` |
| 일반 티켓 상태 | `tickets.status`, 체크인 기록 |
| 좌석 최종 점유 | `seats.status` |
| 재배정 이력 | 티켓의 `seatReassignHistory` |
| 정산 기준 수치 | 주문 수 + 발권 수 + 입장 수 조합 |

## 12. 고령층 친화형 점검표

- 버튼은 명확한 동사 사용
- 숨겨진 제스처 없이 다음 행동이 바로 보일 것
- 숫자, 시간, 좌석 정보는 한 줄 요약으로 반복 표시
- 예매 실패 시 `왜 실패했는지` 문장으로 안내
- 결제 전 마지막 확인 화면에서 공연, 날짜, 좌석, 금액을 큰 글씨로 재확인

## 13. 브랜드 / 오픈 준비

### 브랜드 방향
- 현재 `멜론티켓`은 내부 운영 명칭으로 유지 가능
- 대외 오픈 시 네이버 연계 성격이 드러나는 별도 브랜드 검토
- 공개 티켓, 앱 주문, 네이버 예매를 같은 언어 체계로 통일

### 오픈 전 확정 항목
- 고객 안내 문구
- 취소/환불 정책
- 좌석 공개 정책
- 주문 연결 가이드
- 스태프 운영 가이드
- 트랙 A와 트랙 B 동시 운영 범위

### 동시 운영 원칙
- 트랙 A: 놀티켓 연계형 모바일 발권

- 같은 공연에서도 회차/판매 채널에 따라 둘 다 병행 가능
